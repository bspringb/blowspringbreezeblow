# 콘텐츠 작성 가이드

> 이 문서는 기능이 추가/변경될 때마다 같이 갱신합니다. 실제 코드와 어긋난 부분을 발견하면 문서를 먼저 고치세요.

## 새 글 쓰기

`src/content/blog/` 아래에 `.md` 또는 `.mdx` 파일을 추가합니다. 파일 이름이 곧 URL 슬러그가 됩니다 (`my-post.md` → `/texts/my-post/`). 블로그 URL 프리픽스는 `/texts`이지만(예전엔 `/blog`), 컬렉션 이름과 폴더(`src/content/blog/`)는 내부적으로 그대로 `blog`입니다.

frontmatter 필드:

```yaml
---
title: '글 제목'
description: '목록/검색엔진에 노출될 한 줄 설명'
pubDate: 'Jul 22 2026'
updatedDate: 'Jul 23 2026'      # 선택. 수정한 경우에만
heroImage: ../../assets/blog-placeholder-1.jpg   # 선택
categories: [잡설, 대만 여행]     # 선택. 아래 "카테고리" 참고
relatedObjects: [John Ford]     # 선택. 이 글이 다루는 objects 항목 (아래 "객체" 참고)
---
```

- `title`, `description`, `pubDate`는 필수입니다.
- `categories`를 생략하면 글 상세 페이지 상단 킥커에 "Essay"가 대신 표시되고, 카테고리 리스트 페이지에는 나타나지 않습니다.
- `relatedObjects`를 적으면 글 본문 맨 아래에 구분선(`<hr>`)과 함께 "관련 객체" 목록이 자동으로 표시됩니다(제목 + 타입, 클릭하면 해당 objects 상세 페이지로). 빈 배열이거나 생략하면 이 섹션 자체가 안 뜹니다. 구현은 `src/layouts/BlogPost.astro`에서 `getEntries()`로 참조를 실제 객체 데이터로 해석하는 방식입니다.
- 스키마는 `src/content.config.ts`에서 관리합니다.

## 카테고리

카테고리는 **자유 문자열**입니다 — 고정 목록이나 별도 등록 파일이 없습니다. 글 frontmatter에 원하는 값을 바로 쓰면 그게 카테고리가 됩니다:

```yaml
categories: [영화, 대만 여행]
```

- 그 글은 `/texts/영화`와 `/texts/대만 여행` 리스트 양쪽에 모두 나타납니다.
- "전체글"(`/texts`)은 카테고리가 아니라 필터 없는 기본 목록입니다.
- **새 카테고리를 추가하는 데 코드 수정이 필요 없습니다.** 처음 보는 값을 쓰면 빌드 시 `/texts/<그값>` 페이지가 자동으로 생깁니다.
- 값 자체가 화면에 그대로 보이는 라벨이므로, 오타를 내면(예: 한 글은 `영화`, 다른 글은 `film`) 그 오타도 별도의 카테고리로 갈라져 보입니다 — 표기를 통일해서 쓰세요. 이건 빌드 에러로 안 잡힙니다(자유 문자열이라 유효성 검사 대상이 아님).
- 구현: `src/lib/categories.ts`의 `getAllCategories()`/`getPostsByCategory()`가 전체 글을 스캔해 존재하는 카테고리와 개수를 계산합니다. `CategoryTabs.astro`, `/texts/[category].astro`가 이걸 사용합니다.

## 링크 걸기

마크다운/MDX 문법을 그대로 씁니다. Astro가 별도로 해줄 건 없습니다.

```md
[다른 글 보기](/blowspringbreezeblow/texts/다른글슬러그/)
[외부 사이트](https://example.com)
```

## 객체(objects) 쓰기 — 감독/영화/단어/장소 등 뭐든

scaruffi.com처럼 실체(감독, 영화, 단어, 장소, 여행지...)를 다루는 컬렉션입니다. `blog`와 별개이고, 카테고리 시스템과도 무관하게 동작합니다. **타입마다 컬렉션을 따로 만들지 않습니다** — `objects` 컬렉션 하나에 `type` 필드로 종류를 구분합니다.

`src/content/objects/`에 파일을 추가합니다. 파일 이름(확장자 제외)이 곧 객체 ID이자 URL(`/objects/<id>/`)입니다.

```yaml
---
type: 감독                   # 자유 문자열 — 감독, 영화, 단어, 장소, 여행지 뭐든. 상단 메뉴에 그대로 라벨로 뜨니 한글 추천
title: '감독 이름'
description: '한 줄 소개'   # 선택
attributes:                 # 선택, 자유 키-값. 타입마다 다른 속성을 써도 됩니다
  국가: 브라질               # 값 하나면 스칼라
  활동시기: 1950-현재
  장르: [드라마, 스릴러]      # 값 여러 개면 배열 — 모든 속성이 멀티값 가능
relatedObjects: [다른객체파일명]  # 선택 — 다른 objects 항목을 참조 (예: 영화→감독)
image: ../../assets/xxx.jpg  # 선택
---
본문에는 소개/에세이를 원하는 만큼 길게 씁니다.
```

- **새 객체 타입을 추가하는 데 코드 수정이 필요 없습니다.** `type: 단어`라고 처음 쓰는 파일을 추가하면 그걸로 끝 — `/objects/type/단어` 목록 페이지와 상단 메뉴 항목이 빌드 시 자동으로 생깁니다 (아래 "상단 내비게이션" 참고).
- `attributes`는 완전히 자유 형식이라 타입마다 다른 속성 이름을 써도 됩니다. 대신 오타를 컴파일 타임에 잡아주지는 않습니다.
- **모든 속성은 멀티값이 가능합니다** — 값이 하나면 그냥 스칼라(`연도: 1939`)로, 여러 개면 배열(`장르: [드라마, 스릴러]`)로 씁니다. 기존에 이미 있던 스칼라 값 파일들은 그대로 유효합니다(마이그레이션 불필요). 상세 페이지 표에는 배열 값이 쉼표로 이어져 표시되고, 각 항목이 개별적으로 `relatedObjects`와 매치되면 그 항목만 링크가 됩니다.
- `relatedObjects`/글의 `relatedObjects`에 존재하지 않는 파일명을 적으면(오타 포함) **빌드가 에러로 실패**합니다 — `reference()`가 검증해주기 때문입니다. 값은 대상 파일의 **파일명(확장자 제외)과 정확히 일치**해야 합니다(대소문자·띄어쓰기 포함).
- 객체 상세 페이지(`/objects/<id>/`)에는 관련 객체와 관련 문서가 **양방향으로 자동 표시**됩니다 — A가 B를 `relatedObjects`로 가리키면, B의 페이지에도 A가 뜹니다. 이 로직은 `src/lib/objects.ts`의 `getRelatedObjects`/`getRelatedDocs`가 처리합니다.
- **attributes 테이블에 어떤 행을 보여줄지는 항상 `attributes`에 직접 적은 키만 대상입니다** (모든 `relatedObjects`가 자동으로 행이 되지는 않습니다 — 관련 객체가 여러 개일 때 원치 않는 행이 줄줄이 생기는 걸 막기 위해서입니다). 다만 그 값이 `relatedObjects`로 연결된 객체의 제목과 같으면 **링크 연결만 자동**으로 됩니다 (예: `attributes: {감독: 존 포드}` + `relatedObjects: [John Ford]` → "감독" 행이 John Ford 페이지로 링크). 즉 "무엇을 보여줄지"는 사람이 고르고, "링크로 만들지"만 자동화되어 있습니다. 이 로직은 `src/layouts/ObjectPage.astro`의 `attributeRows` 계산부에 있습니다.
- 블로그 글에서도 `relatedObjects: [파일명]`으로 객체를 연결할 수 있습니다 — 카테고리와 별개로, 연결한 객체의 "관련 문서" 목록에 나타납니다.

`src/content/objects/example-*.md`는 이 구조가 실제로 동작하는지 확인하기 위한 테스트 데이터입니다(감독/영화 페어가 두 세트, `word`/`place` 타입 예시 포함). 자유롭게 수정하거나 지우세요.

## 콘텐츠 입력 폼 (프론트매터 손으로 안 치고 입력하기)

`npm run admin`으로 로컬 전용 입력 폼(`http://localhost:4322`)을 띄울 수 있습니다. `scripts/admin/server.mjs`(작은 Node HTTP 서버, `js-yaml`로 frontmatter 파싱/직렬화)와 `scripts/admin/public/index.html`(순수 HTML/JS 폼)로 이루어져 있고, **Astro 빌드·배포와는 완전히 분리**되어 있습니다 — `astro.config.mjs`나 `dist/` 산출물에 아무 영향이 없고, GitHub Pages에도 포함되지 않습니다. `npm run dev`(Astro)와 동시에 켜둬도 됩니다(포트가 다름, 4322).

폼 위쪽의 **컬렉션 선택**(objects / blog)으로 어느 걸 편집할지 고릅니다. 공통 동작:

- **새로 만들기**: 필드를 채우고 저장하면 해당 컬렉션 폴더(`src/content/objects/` 또는 `src/content/blog/`)에 `<id>.md`가 새로 생깁니다. 같은 이름의 파일이 이미 있으면 저장을 막습니다(덮어쓰기 방지).
- **기존 항목 불러와 수정**: 드롭다운에서 기존 객체/글을 고르면 폼에 값이 채워지고, 저장하면 같은 파일을 덮어씁니다. **파일명(=참조 id)을 바꿔서 저장하는 기능은 없습니다** — 이름이 바뀌면 다른 곳의 `relatedObjects` 참조나 URL 슬러그가 깨지기 때문에, 리네임이 필요하면 지금처럼 직접 파일명을 바꾸고 참조도 손으로 맞추세요.

**objects 폼**:
- **attributes 행**: 키 입력창에 커서를 두면, **같은 타입의 기존 객체들이 실제로 썼던 키**가 자동완성으로 뜹니다(`/api/meta`가 타입별로 모아서 내려줌). 값은 카테고리 입력과 같은 방식(검색 + 클릭해서 칩으로 추가, 없으면 새로 입력)이라 **여러 개를 한 속성에 넣을 수 있습니다** — 하나만 넣으면 스칼라로, 여러 개 넣으면 배열로 저장됩니다. 새로 만들기 모드에서 타입을 지정하면 그 타입에 등록된 적 있는 키가 전부 빈 값으로 깔리고, 필요 없는 키는 ✕로 빼면 됩니다.
- **relatedObjects**: 제목으로 검색해서 고르는 방식이라, 오타로 존재하지 않는 파일명을 적어서 빌드가 깨지는 일이 원천 차단됩니다.

**blog 폼**:
- `pubDate`/`updatedDate`는 날짜 선택창(기본값: 오늘)이고, 저장 시 `YYYY-MM-DD` 형식으로 들어갑니다.
- `categories`는 기존에 쓰인 카테고리를 검색해서 고르거나, 없으면 그 자리에서 "새로 추가"를 눌러 새 카테고리 문자열을 바로 만들 수 있습니다(코드 수정 없이 새 카테고리가 되는 자유문자열 구조와 동일).
- `relatedObjects`는 objects 폼과 마찬가지로 제목 검색 후 선택.
- `heroImage`는 실제 이미지 파일 업로드는 지원하지 않습니다 — 이미지를 `src/assets/`에 직접 넣고 상대경로(`../../assets/xxx.jpg`)만 입력하세요.

두 컬렉션 모두 `/api/objects`, `/api/blog` 아래 REST 스타일 엔드포인트(`GET .../meta`, `GET .../:id`, `POST ...`)로 동작하며, 같은 패턴을 따르므로 나중에 다른 컬렉션이 생겨도 이 구조를 그대로 복제해서 확장할 수 있습니다.

## 홈화면 구조 (사이드바 + 카테고리 그리드)

홈(`/`)은 정적 목록이 아니라 상태를 가진 대시보드입니다 — React 아일랜드 두 개가 `nanostores`로 상태를 공유합니다.

- `src/stores/pinnedCategories.ts`: 사이드바에 "고정"된 카테고리 이름 배열(클릭 순서). `pin()`/`unpin()`.
- `src/components/react/CategoryGrid.tsx`: 카테고리별 블록(2~3열), 각 블록의 "+"가 위 스토어에 `pin()`.
- `src/components/react/Sidebar.tsx`: 스토어를 구독 — 고정된 게 없으면 "전체글", 있으면 고정된 카테고리들을 클릭 순서대로 쌓아 보여줌(x로 `unpin()`).
- 데이터(글 목록, 카테고리별 글)는 전부 `src/pages/index.astro`에서 빌드 타임에 계산해 두 컴포넌트에 props로 넘깁니다 — 컴포넌트들은 화면 상태(정렬 방향, 고정 여부)만 다루고 클라이언트에서 다시 데이터를 가져오지 않습니다.
- 새 화면 상태를 이 홈화면에 추가할 때도 이 패턴(store + 여러 아일랜드가 구독)을 따르세요.

## 상단 내비게이션 (최상위 객체 타입 + 블라인드)

`src/components/Header.astro`는 로고/소셜링크만 담당합니다. 내비게이션은 `src/components/NavLinks.astro`가 그리는데, 구성이 두 부분으로 나뉩니다:

- **`Texts`** — 블로그 글 목록(`/texts`)으로 가는 고정 링크. 블라인드 바깥에 항상 보입니다.
- **객체 타입 블라인드** — `src/lib/objectNav.ts`의 `getNavTypeGroups()`가 `objects` 컬렉션에 실제로 존재하는 `type` 값들을 모아 최상위 항목으로 나열합니다. **새 객체 타입을 추가하면 코드 수정 없이 이 메뉴에도 자동으로 뜹니다.**
- 이 묶음 설정(`TYPE_CHILDREN`/`VIRTUAL_GROUPS`)은 코드가 아니라 **`src/data/navGroups.json`** 데이터 파일에 있고, `objectNav.ts`가 그걸 그대로 읽습니다 — 코드를 직접 안 고쳐도 되고, **관리자 폼(`npm run admin`)의 "설정 (내비게이션)" 탭에서 편집할 수 있습니다.**
- 특정 타입을 다른 **실제 타입** 아래로 묶고 싶을 때(예: "감독"은 "영화" 아래에만 — "영화" 자신도 실제 객체가 있는 타입), `navGroups.json`의 `typeChildren`에 한 줄 추가하면 됩니다:
  ```json
  "typeChildren": {
    "영화": ["감독"]
  }
  ```
  여기 등록된 타입(`감독`)은 최상위 메뉴에 따로 뜨지 않고, 부모(`영화`)에 마우스를 올렸을 때 뜨는 팝업에만 나타납니다.
- 반대로 부모 라벨 자체는 **실제 타입이 아니고 화면에만 존재하는 묶음**이고 싶을 때(예: "음악"이라는 객체는 없고, 그 아래 "아티스트"/"앨범"/"곡"만 실제 타입인 경우), `virtualGroups`를 씁니다:
  ```json
  "virtualGroups": {
    "음악": ["아티스트", "앨범", "곡"]
  }
  ```
  소속 타입 중 실제 객체가 하나도 없으면 이 그룹 자체가 메뉴에서 안 보입니다. **주의**: `virtualGroups`의 라벨 이름과 같은 문자열을 실제 `type:`으로 쓰면 메뉴에 같은 이름이 두 번 뜹니다 — 라벨은 실존하지 않는 이름으로 고르세요.
- 관리자 폼의 "설정" 탭에서는 부모/라벨 이름을 직접 타이핑하고, 자식/소속 타입은 기존 타입 목록에서 검색해 칩으로 추가합니다(카테고리·relatedObjects와 같은 방식). 저장하면 `src/data/navGroups.json`을 바로 덮어씁니다.
- **헤더(최상위 메뉴 텍스트)와 드롭다운 안의 "자기 자신" 링크는 서로 다른 곳으로 갑니다** — 자식이 있는 항목은 헤더를 누르면 부모+자식을 모두 합친 목록(`/objects/group/<라벨>`, `src/pages/objects/group/[group].astro`, `getGroupMemberTypes()`가 라벨→합칠 타입 목록을 계산)으로 가고, 드롭다운 팝업 안에서 "자기 자신"만 따로 누르면 그 타입 하나만 있는 `/objects/type/<타입>`으로 갑니다. 예: "영화" 헤더 → 영화+감독을 합친 목록, 드롭다운의 "영화" → 영화만. 가상 그룹(음악)은 자기 타입이 없으므로 드롭다운에 "자기 자신" 링크가 아예 없고 소속 타입만 나열됩니다.
- 객체 `type` 값은 화면에 그대로 라벨로 쓰이므로, 카테고리와 마찬가지로 **한글로 쓰는 걸 추천합니다** (`type: 영화`, `type: 감독`처럼 — `type: film`이라고 쓰면 메뉴에 영어 그대로 뜹니다).

이 블라인드는 항목이 늘어나 한 줄을 넘길 때를 대비한 것입니다(`NavLinks.astro`의 진짜 우선순위 내비게이션 구현):

- `.nav-links`는 `flex-wrap: nowrap` + `overflow: hidden`으로 한 줄만 보입니다. 컴포넌트에 포함된 스크립트가 로드/리사이즈 시마다 모든 항목을 원래 자리로 되돌린 뒤, `getBoundingClientRect()`로 각 항목이 "펼치기" 버튼 자리를 침범하는지 측정합니다.
- **실제로 넘치는 항목만** DOM에서 통째로 옮겨져(복제 아님) `#nav-overflow`에 들어갑니다 — 안 넘치는 항목은 항상 원래 자리에 그대로 남아, 펼치기 전/후로 첫 줄 모양이 절대 변하지 않습니다.
- "펼치기" 버튼은 넘치는 항목이 하나도 없으면 자동으로 숨겨집니다. 누르면 `#nav-overflow`가 검색창 자동완성처럼 헤더 아래에 드롭다운으로 펼쳐집니다(한 줄에 가로로 나열, 넘치면 다음 줄로).

## 나중에 또 다른 새 "섹션"(컬렉션)을 추가하려면

`objects`가 정확히 이 패턴의 실제 예시입니다. `blog`/`objects`와 성격이 완전히 다른 컨텐츠(예: 댓글, 갤러리)를 새로 만들고 싶다면 같은 방식을 따르세요.

1. **콘텐츠 폴더 + 스키마 추가** (`src/content.config.ts`) — `objects` 정의를 참고하세요.
2. **컬렉션 간 링크는 `reference()`로.** 참조 대상 ID가 실제로 존재하는지 빌드 타임에 검증됩니다.
3. **리스트/상세 페이지는 `src/pages/objects/`처럼 `getStaticPaths` 기반 공용 템플릿으로.** 타입/카테고리를 가리지 않는 제네릭한 페이지 하나로 여러 종류를 처리하는 쪽이 낫습니다.

이 문서도 함께 갱신하는 것 잊지 마세요.
